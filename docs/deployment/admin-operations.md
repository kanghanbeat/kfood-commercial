# Admin Operations Runbook

## Admin Creation

For MVP, create admin users through Supabase Dashboard invite or manual user creation. Public sign-up must never grant admin privileges. Client-provided role values, user-editable metadata, and form input are not trusted authorization sources.

Recommended role source:

- Use `profiles.role` for app-level authorization.
- Use server-issued `app_metadata` claims later only if generated by trusted backend code.

## Safe Role Assignment SQL Template

Use placeholders only. Verify the target user before running.

```sql
begin;

-- Replace with the exact invited admin email.
with target_user as (
  select id
  from auth.users
  where email = 'target-admin-email@example.com'
  limit 1
)
update public.profiles
set role = 'admin'
where id = (select id from target_user)
  and role in ('user', 'reviewer', 'moderator', 'admin');

commit;
```

Expected role values: `user`, `reviewer`, `moderator`, `admin`.

## Admin Route Policy

- If not logged in, redirect to login.
- If logged in but not admin or moderator, redirect to home or show a forbidden screen.
- If `EXPO_PUBLIC_ADMIN_ROUTE_ENABLED=false`, hide admin navigation and block direct admin route entry.
- Do not rely on client-side navigation alone; Supabase RLS and RPC checks must enforce permissions.

## Audit Policy

Log these admin actions:

- Label approvals and rejections.
- Content hides or restores.
- User suspensions.
- Role changes.
- Manual point adjustments.

Never log secret values, access tokens, refresh tokens, API keys, authorization headers, passwords, or service-role keys.
